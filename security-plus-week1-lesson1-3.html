Chapter 1.3 — Domain 1.0: General Security Concepts

Public Key
Infrastructure (PKI)

The operational system that makes public key cryptography work at scale. Certificate anatomy, trust chain validation, revocation checking, and certificate types. Expect 4 to 6 questions on the exam.

Lesson 1.3 Domain 1.0 — 12% 13 Days to Exam Est. 90 min
Overview
Why This Lesson Matters

PKI is the operational system behind every HTTPS connection, every code-signed application, every S/MIME email, and every VPN certificate. The exam tests PKI in three ways: certificate anatomy, trust chain validation, and revocation checking. Expect 4 to 6 questions across these areas.

What Carries Over From Chapter 1.2

Asymmetric encryption, digital signatures, hashing, and the four cryptographic goals. This lesson builds the trust system on top of those primitives.

Chapter 01The Trust Problem PKI Solves

Asymmetric cryptography lets anyone publish a public key and receive encrypted messages or verified signatures. Nothing about the math proves the public key actually belongs to the claimed owner.

If an attacker publishes a public key claiming to be your bank, and you use it, the attacker decrypts everything you send. The cryptography works. The trust fails.

PKI solves this by introducing a trusted third party (the Certificate Authority) that vouches for the binding between an identity and a public key. That vouching takes the form of a signed digital certificate.

Exam Trap: CN vs. SAN

Modern browsers reject certificates that rely on Common Name alone. The Subject Alternative Name (SAN) extension is required for TLS server certificates as of Chrome 58 (2017). If a scenario describes a certificate error where the browser rejects a cert that lists the domain in CN but not in SAN, this is the cause.

Chapter 03The Certificate Authority Hierarchy

CAs are organized in a tree. Trust flows from the top down.

CA Hierarchy [ Root CA ] Self-signed, offline, trust anchor | | signs v [ Intermediate CA ] Online, signs subscriber certs | | signs v [ Leaf Certificate ] The cert on a web server, VPN, etc. Subject: yourbank.com

Why Intermediate CAs Exist

  • The root CA is kept offline in an HSM to protect the trust anchor
  • If an intermediate CA is compromised, only that intermediate is revoked. The root and other intermediates continue functioning
  • Different intermediates can be dedicated to different purposes: TLS server auth, code signing, S/MIME
Trust Anchor Mechanism

Your operating system and browser ship with a preloaded list of trusted root CAs (the trust store). Windows, macOS, iOS, Android, and each major browser maintain their own trust stores. When your browser validates a certificate, it walks up the chain until it reaches a root that exists in the trust store. If it does, the chain is trusted. If it does not, the connection fails.

Chapter 04Chain Validation Step by Step

When you visit https://yourbank.com, your browser performs these checks in order.

Chain Validation Sequence Step 1: Server presents its leaf certificate + intermediate cert(s) Step 2: Browser extracts the leaf's Issuer field Step 3: Browser finds the intermediate cert matching that Issuer Step 4: Browser verifies leaf's signature using intermediate's public key Step 5: Browser extracts the intermediate's Issuer field Step 6: Browser looks up that Issuer in its local trust store Step 7: If found, verifies intermediate's signature using root's public key Step 8: Browser checks validity dates on every cert in the chain Step 9: Browser checks revocation status on the leaf Step 10: Browser verifies Subject or SAN matches the requested hostname Step 11: If all checks pass, connection proceeds with the leaf's public key
Every Step Must Pass

A single failure breaks the chain. The exam tests this by describing a specific failure and asking which step caused it.

Common Chain Failures

FailureCause
Untrusted issuerRoot is not in the browser's trust store (self-signed certs fail here)
Expired certificateCurrent date is outside the validity window
Hostname mismatchCertificate is valid but does not cover the requested domain
Missing intermediateServer did not send the intermediate cert; browser cannot build the chain
Revoked certificateCRL or OCSP returns "revoked"

Chapter 05Revocation: CRL, OCSP, and OCSP Stapling

Certificates have expiration dates, but a CA may need to invalidate a cert before expiration. Reasons include private key compromise, employee termination, or certificate misissuance. Three mechanisms exist.

CRL (Certificate Revocation List)

PropertyValue
MechanismClient downloads entire signed list of revoked serial numbers
LatencyHigh. CRLs published hours or days after revocation
BandwidthHigh. Full CRL download can be megabytes
PrivacyBetter. Client does not tell the CA which specific cert it is checking
FreshnessPoor. Cached CRLs may be outdated

OCSP (Online Certificate Status Protocol)

PropertyValue
MechanismClient sends serial number to OCSP responder; gets "good", "revoked", or "unknown"
LatencyLow. Real-time query
BandwidthLow. Single request-response per cert
PrivacyPoor. CA learns exactly which sites you visit
FreshnessGood. Responses current at query time

OCSP Stapling

PropertyValue
MechanismWeb server queries OCSP responder itself; staples response to TLS handshake
LatencyVery low. No extra client-side query
BandwidthVery low. Response is part of the handshake
PrivacyExcellent. Client never contacts the CA
FreshnessGood. Server refreshes stapled response every few hours
Exam Rule

If a scenario emphasizes user privacy, low latency, and current revocation status simultaneously, the answer is OCSP Stapling. It is the best of all three worlds.

Certificate Pinning

Client hardcodes the expected certificate (or public key) for a specific server. Any certificate not matching the pin is rejected, even if the chain would otherwise validate. Used heavily in mobile apps to defeat MITM attacks that rely on rogue CAs.

Pinning Tradeoff

Pinning breaks when the legitimate certificate is rotated. Requires app updates or a pin refresh mechanism.

Wildcard Limitations the Exam Tests

*.example.com covers mail.example.com but does NOT cover example.com (the apex) or secure.mail.example.com (multi-level subdomain). The wildcard replaces exactly one label at the leftmost position.

Chapter 07Self-Signed Certificates and Private CAs

Self-Signed Certificate
  • Signs itself. No CA involved
  • Issuer and Subject fields are identical
  • Trusted only if manually added to client trust store
  • Common for internal test environments and IoT devices
  • Browsers show a security warning on public web
Private (Internal) CA
  • Organization runs its own CA hierarchy
  • Root pushed to every corporate device via GPO or MDM
  • Used for internal HTTPS, RDP, VPNs, 802.1X
  • Not trusted outside the organization
  • Scales to thousands of certs
Exam Trap

A scenario describes a company that wants to issue TLS certificates to internal servers without paying a public CA. The correct answer is a private CA, not self-signed certificates. Self-signed does not scale beyond a handful of devices.

Q & A

Attempt every answer before revealing.

Question 01 of 04

A user visits https://internal.company.com from their corporate laptop and gets no warning. They visit the same URL from their personal laptop and get a "This connection is not secure" error. The certificate is identical in both cases.

Explain the technical reason.

Answer

The company operates a private (internal) CA. Its root certificate is installed in the trust store of every corporate laptop via Group Policy, MDM, or equivalent tooling. When the corporate laptop validates the certificate chain, it walks up to the private root, finds it in the local trust store, and trusts the chain.

The personal laptop has no such installation. When it walks the chain, it reaches the private root and cannot find it in the trust store. Chain validation fails, and the browser displays the warning.

The certificate itself is valid. Trust is a function of the local trust store, not the certificate.

Question 02 of 04

A large e-commerce site uses OCSP for revocation checking. Their CTO raises a privacy concern in a board meeting.

What is the concern, and what specific technology addresses it without abandoning real-time revocation checking?

Answer

Privacy concern: Every time a user visits the e-commerce site, the user's browser sends an OCSP query to the CA. The CA learns which certificates users are checking, the approximate time of each visit, and the user's IP address. Over time, the CA can build a profile of which sites individual users visit.

Solution: OCSP Stapling. The e-commerce web server queries the OCSP responder itself, receives a signed OCSP response, and attaches it to the TLS handshake. The user's browser gets the revocation status from the server, not from the CA. The CA sees only the server's query, not individual user queries. Real-time revocation status is preserved.

Question 03 of 04

A developer purchases a wildcard certificate for *.example.com and deploys it to mail.example.com, api.example.com, and example.com (the apex domain). Users report certificate errors when visiting example.com directly but not the subdomains.

Why? How do you fix it?

Answer

A wildcard certificate for *.example.com covers exactly one label at the leftmost position. It matches mail.example.com and api.example.com because the wildcard substitutes for mail and api.

It does NOT match example.com (the apex domain) because there is no label to substitute for. The wildcard requires something in that leftmost position.

Fix: Purchase a certificate that covers both the wildcard and the apex, or use a SAN certificate that explicitly lists example.com and *.example.com as separate entries.

Question 04 of 04

A corporate security team discovers that a signing key for their internal code-signing certificate was exposed in a public GitHub repository 4 hours ago.

Walk through the immediate technical response, in order, and explain the risk if any step is delayed.

Answer

Step 1: Revoke the certificate immediately. Submit a revocation request to the issuing CA. The CA updates the CRL and OCSP responder.

Step 2: Verify OCSP responder returns "revoked". Query directly to confirm propagation.

Step 3: Rotate any secrets and access tokens in the same repository. A leaked signing key rarely leaks alone.

Step 4: Identify all software previously signed with the compromised key. Every artifact signed with that key is now suspect.

Step 5: Reissue new signing certificates using new key material generated inside an HSM. The old key is dead; the new key must never leave secure hardware.

Step 6: Re-sign trusted artifacts and publish updated versions.

Step 7: If applicable, add the compromised certificate to distrust lists (Windows Authenticode Distrust List, Apple's revocation mechanism). Revocation checking is not always enforced by every client. Explicit distrust ensures rejection.

Risk of delay: Every hour the compromised key remains valid is another hour an attacker can sign malicious code that will pass certificate validation on unpatched systems. Once the attacker signs a malware sample, that sample retains its signature validity indefinitely on any client that does not check revocation status. Some Windows clients cache OCSP responses for hours or do not enforce revocation for code signing certificates at all, extending the attacker's window.
