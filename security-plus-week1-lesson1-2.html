Chapter 1.2 — Domain 1.0: General Security Concepts

Cryptography
Foundations

Symmetric and asymmetric algorithms, hashing, digital signatures, and the four goals of cryptography. Expect 6 to 10 questions on the exam. Most are scenario-based.

Lesson 1.2 Domain 1.0 — 12% 13 Days to Exam Est. 90 min
Overview
Why This Lesson Matters

Cryptography questions on SY0-701 are not math questions. They test whether you know which algorithm applies to which use case, which are still safe, and which are deprecated. You will see 6 to 10 crypto questions on the exam. Most are scenario-based, and most candidates lose points here by picking a broken algorithm or confusing which key does what.

What Carries Over From Network+

IPSec AH vs ESP, WPA3 SAE, Perfect Forward Secrecy at a surface level. This lesson goes deeper on the primitives underneath: symmetric vs asymmetric algorithms, hashing, digital signatures, and the four security goals each mechanism satisfies.

Chapter 01The Four Goals of Cryptography

Every crypto question on the exam maps to one of these four goals. Learn them in this order.

GoalWhat It ProvidesHow It Is Achieved
ConfidentialityData cannot be read by unauthorized partiesEncryption (symmetric or asymmetric)
IntegrityData has not been alteredHashing, HMAC
AuthenticityThe sender is who they claim to beDigital signatures, MAC
Non-repudiationThe sender cannot later deny sendingDigital signatures (asymmetric only)
Critical Distinction: Hash vs. HMAC vs. Signature

A hash provides integrity only. It does not prove who created the data. To prove both integrity and authenticity, use an HMAC or a digital signature. Non-repudiation requires asymmetric cryptography because the private key is unique to the signer.

Chapter 02Symmetric Encryption

Symmetric encryption uses one shared key for both encryption and decryption. Both parties must have the same key.

Key Properties

  • Fast. 100 to 1000 times faster than asymmetric.
  • Requires secure key exchange before communication begins.
  • Scales poorly. For N parties, you need N(N-1)/2 keys.

Algorithms You Must Know

AlgorithmKey SizesStatusNotes
AES128, 192, 256 bitCurrent standardUsed in TLS 1.3, WPA3, disk encryption
3DES168 bit effectiveDeprecated (2023)Do not select on exam scenarios
DES56 bitBrokenNever correct on any exam question
RC4VariableBrokenHistorically used in WEP and old TLS
ChaCha20256 bitCurrent standardMobile-friendly alternative to AES in TLS 1.3
Exam Rule

If a scenario asks for symmetric encryption today, the answer is AES-256 or ChaCha20. Anything else is either deprecated or a distractor.

AES Modes of Operation

AES is a block cipher. Modes define how blocks are chained together.

ModeFull NameProvidesNotes
ECBElectronic CodebookConfidentiality onlyInsecure. Identical plaintext blocks produce identical ciphertext. Never use.
CBCCipher Block ChainingConfidentiality onlyRequires an initialization vector (IV). Vulnerable to padding oracle attacks.
GCMGalois/Counter ModeConfidentiality + Integrity + AuthenticityPreferred for TLS 1.3. Provides authenticated encryption (AEAD).
CCMCounter with CBC-MACConfidentiality + IntegrityUsed in WPA2.
Exam Scenario Tip

If the question mentions "authenticated encryption" or "AEAD," the answer is AES-GCM or ChaCha20-Poly1305.

Chapter 03Asymmetric Encryption

Asymmetric encryption uses two mathematically related keys: a public key (shared openly) and a private key (kept secret). Anything encrypted with one can only be decrypted with the other.

The Two Directions

Understanding which direction is used for what is a common exam trap.

Encrypt with Recipient's Public Key
  • Goal: Confidentiality
  • Only the recipient's private key can decrypt
  • Anyone can send an encrypted message to the recipient
  • Example: sending an encrypted email
Sign with Sender's Private Key
  • Goal: Authenticity + Non-repudiation
  • Anyone can verify with the sender's public key
  • Only the sender could have produced the signature
  • Example: signed software updates
Exam Rule

You encrypt to someone using their public key. You sign as yourself using your private key. Get this backwards and you will miss every asymmetric scenario question.

Algorithms You Must Know

AlgorithmPurposeEquivalent Security Size
RSAEncryption, signatures, key exchange2048 bit minimum, 3072 recommended
ECCSignatures, key exchange256 bit (equivalent to 3072 RSA)
DSASignatures only2048 bit; being phased out
ECDSASignatures using elliptic curves256 bit
Diffie-Hellman (DH)Key exchange only, no encryption2048 bit minimum
ECDH / ECDHEElliptic curve key exchange256 bit; ECDHE provides PFS
Exam Scenario Tip

ECC provides the same security as RSA at a much smaller key size. That is why mobile devices and IoT use ECC. If a scenario mentions constrained devices or battery life, ECC is the answer.

Chapter 04Hybrid Encryption (How TLS Actually Works)

Neither symmetric nor asymmetric alone is practical for real communication. TLS combines both.

TLS Handshake (Simplified) Step 1: Client connects to server Step 2: Server sends its certificate (contains public key) Step 3: Client and server perform key exchange (ECDHE) Produces a shared session key Session key is a symmetric AES key Step 4: All further communication uses AES-GCM with the session key Fast symmetric encryption for the actual data The slow asymmetric operation ran only once, at handshake
Why This Design

Asymmetric solves the "how do we share a key over an untrusted network" problem. Symmetric handles the bulk data transfer because it is fast enough. The session key exists only for that connection. This pattern powers TLS 1.3, IPSec IKE, SSH, and every modern encrypted protocol.

Chapter 05Hashing

A hash function takes any input and produces a fixed-size output (the digest). Hashing is one-way. You cannot reverse a hash to recover the input.

Properties of a Cryptographic Hash

  • Deterministic: same input always produces the same output
  • Fixed output size: SHA-256 always produces 256 bits regardless of input
  • One-way: computationally infeasible to reverse
  • Collision-resistant: computationally infeasible to find two inputs with the same output
  • Avalanche effect: a single-bit input change produces a completely different output
AlgorithmDigest SizeStatus
MD5128 bitBroken. Collision attacks demonstrated in 2004. Do not use for security.
SHA-1160 bitBroken. Collision attack demonstrated in 2017 (SHAttered).
SHA-256256 bitCurrent standard. Part of SHA-2 family.
SHA-384 / SHA-512384 / 512 bitCurrent standard. Stronger variants of SHA-2.
SHA-3VariableCurrent standard. Different internal design than SHA-2.
Exam Rule

If a scenario asks for a hash today, the answer is SHA-256 or stronger. MD5 and SHA-1 are only correct as "what algorithm is broken" answers.

Chapter 06Digital Signatures

A digital signature provides authenticity, integrity, and non-repudiation.

Signing (sender side) 1. Sender hashes the message -> produces digest 2. Sender encrypts the digest with -> produces signature their PRIVATE key 3. Sender attaches signature to message Verifying (recipient side) 1. Recipient hashes received message -> produces digest A 2. Recipient decrypts the signature with -> produces digest B the sender's PUBLIC key 3. If digest A == digest B, signature is valid
What Each Part Proves

If digest A matches digest B, the message was not altered (integrity). Only the sender's private key could have produced a signature that decrypts correctly with the sender's public key (authenticity). The sender cannot deny sending because only they possess the private key (non-repudiation).

Chapter 07Salting, Peppering, and Key Stretching

Hashing a raw password is not enough. Attackers precompute hashes of common passwords (rainbow tables) and look up matches.

Salt
  • A random value added to each password before hashing
  • Stored alongside the hash in the database
  • Two users with the same password get different hashes
  • Defeats rainbow tables completely
Pepper
  • A secret value added to every password before hashing
  • Stored separately from the database (HSM or config)
  • If DB is stolen but pepper is not, hashes stay unrecoverable
  • Adds a layer beyond salting

Key Stretching Algorithms

AlgorithmNotes
bcryptWidely used; cost factor tunable
PBKDF2NIST-approved; used in WPA2, disk encryption
Argon2Winner of the 2015 Password Hashing Competition; current best practice
scryptMemory-hard; resists GPU-based attacks
Exam Rule

If a scenario asks about password storage, the answer involves bcrypt, PBKDF2, Argon2, or scrypt combined with a salt. Never a plain hash. Never encryption.

Q & A

Attempt every answer before revealing. Write your reasoning out loud.

Question 01 of 04

A developer stores user passwords by hashing them with SHA-256 and storing the digest in the database. No salt is used. What are the two specific weaknesses in this design?

Answer

Weakness 1: No salt. Two users with the same password produce identical hashes. An attacker who compromises the database can identify shared passwords instantly. Rainbow table attacks work because precomputed hash tables of common passwords match directly against the stored hashes.

Weakness 2: SHA-256 is too fast for password storage. SHA-256 is designed to be fast for file integrity checks. For passwords, this is the opposite of what you want. An attacker with a GPU can compute billions of SHA-256 hashes per second in a brute-force attack. Use a slow hash designed for passwords: bcrypt, Argon2, PBKDF2, or scrypt, combined with a per-user salt.

Question 02 of 04

An employee sends an encrypted email to a colleague using the colleague's public key. The colleague receives it, decrypts it, and reads the contents. Which cryptographic goal is satisfied? Which is not? What would you add to satisfy the missing goal?

Answer

Satisfied: Confidentiality. Only the recipient's private key can decrypt the message. The message is unreadable to anyone else.

Not satisfied: Authenticity and Non-repudiation. The recipient has no way to verify the sender is actually the person they claim to be. Anyone with access to the colleague's public key (which is public) could have sent the encrypted message.

Solution: Add a digital signature. The sender hashes the message and encrypts the hash with their own private key. The recipient verifies the signature with the sender's public key. Now the recipient has confidentiality (from asymmetric encryption to the recipient) and authenticity plus non-repudiation (from the signature by the sender).

Question 03 of 04

A security team is choosing an encryption algorithm for an IoT sensor network. The sensors have limited battery, minimal CPU, and constrained memory. They need to negotiate session keys with a central gateway over an untrusted network.

Which algorithm family is best suited and why? Name specific algorithms for key exchange, signatures, and bulk encryption.

Answer

Family: Elliptic Curve Cryptography (ECC). ECC provides equivalent security to RSA at much smaller key sizes. A 256-bit ECC key provides roughly the same security as a 3072-bit RSA key. Smaller keys mean less memory, less computation, less battery drain, faster handshakes.

Key exchange: ECDHE (Elliptic Curve Diffie-Hellman Ephemeral). Provides Perfect Forward Secrecy because a new session key is generated for each connection. If a long-term key is later compromised, past sessions remain protected.

Signatures: ECDSA. Elliptic curve digital signature algorithm.

Bulk encryption: ChaCha20-Poly1305 (mobile-optimized) or AES-128-GCM for the actual data after handshake.

Question 04 of 04

A financial institution is subject to a compliance requirement that says "all archived transaction records must be verifiable years after they are written, and the writer must not be able to deny writing them, even if the writer leaves the company."

Which cryptographic mechanism satisfies this requirement? What operational risk must be managed?

Answer

Mechanism: Digital signature. Each transaction record is signed using the writer's private key at the time of writing. The signature provides integrity (record was not altered), authenticity (only that writer's key could have produced this signature), and non-repudiation (the writer cannot deny it).

Operational risk: Private key lifecycle. The private key must be protected during its active life, and the signature must remain verifiable long after the key expires or the employee leaves. Solutions include:

1. Store all signing keys in an HSM with strict access controls and audit logging.
2. Retain the corresponding public key and certificate chain in an archive so signatures can be verified after key expiration.
3. Timestamp signatures using a trusted timestamping authority so verification proves the signature existed at the time of writing, even if the certificate later expires or is revoked.

Without timestamping, a signature that verifies today may fail to verify in 5 years when the certificate has expired, creating a compliance failure.

Recall Summary
Quick Recall

Symmetric (fast, shared key): AES-256, ChaCha20. Mode matters: GCM for authenticated encryption.
Asymmetric (slow, key pair): RSA-2048+, ECC-256. Encrypt with recipient public. Sign with sender private.
Hashing (integrity only): SHA-256, SHA-3. Never MD5 or SHA-1.
Digital signatures (integrity + authenticity + non-repudiation): Hash then sign the hash with private key.
Password storage: bcrypt, Argon2, PBKDF2, or scrypt with per-user salt.

TLS 1.3 in Practice

Key exchange: ECDHE (provides PFS)
Authentication: RSA or ECDSA certificate
Bulk encryption: AES-GCM or ChaCha20-Poly1305
Hashing: SHA-256 or SHA-384
